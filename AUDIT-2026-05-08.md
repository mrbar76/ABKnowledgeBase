## 13. LIVE-DATA FINDINGS (from design agent's API audit)

These were surfaced by hitting the live API directly today; they are about *content shape*, not code structure.

### 13.1 `/briefing` is dumping the database, not curating it

`/briefing` returns a markdown wall containing all 19 overdue tasks, all 46 stale tasks, all 13 waiting-on items, full plan rationale, yesterday's completion notes, gamification rings, and recovery components. **That's an export, not a "today" surface.**

The Today tab calls `?format=json`, which is better-shaped, but it still inherits the same "list everything" instinct. The redesign needs an explicit cap (e.g. top-3 hero, max-5 list) and triage (prioritize what *changed* over what's *stale*).

Code reference: `routes/briefing.js:509` (the GET handler) and `generateBriefing()` helper.

### 13.2 "Top 3 Focus" has no intelligence

Ranker is `(overdue, priority)` ascending. Today it returns 2 work tasks + 1 work blocker. **Family is invisible. Spartan is invisible. The current ranker is single-pillar by accident.**

Code reference: `routes/briefing.js:19-21,32-34,41-43` — the score formula. No pillar diversification, no recency boost, no "what got done yesterday" awareness.

### 13.3 Raw data is leaking to the surface

Hard requirement violations seen live:

| Field | Example value | Where it leaks |
|-------|---------------|----------------|
| Goal `anchor_source` | `"workout ac0407a7, working set 3 = 175×5 RPE ~8 clean. Per spec section 6"` | Goal detail (when opened) |
| Workout `focus` | `"rdl_pull_grip"` (slug, not human) | Training Today, History |
| Task title | `"PROMPT AVI: Pick Mother's Day NYC activity"` | Productivity tabs |
| Task title | `"[WAITING ON: Name] ..."` × 13 tasks | Productivity, Today |

The `db.js` boot migration strips `PROMPT AVI:` and `[WAITING ON:]` prefixes, but **it only catches the literal prefix forms** — anything else still leaks. Write a presentation-layer cleanup helper in `app.js` (`cleanTaskTitle()` exists at 3782 but isn't applied universally).

### 13.4 Voice violation in the *data layer*, not just UI

Plan rationale field contains: `"REVISED v3 after Avi pushback. Goal 2 ... was being missed by the v2 RDL substitution"`. That's an internal commit message rendered as user-facing text.

Em-dashes throughout the data layer (e.g. `"Strength A — Deadlift Re-entry"`). The briefing-markdown writer is not in coach voice. Either the LLM prompt that writes plans needs voice constraints, or the renderer strips em-dashes on the way out.

### 13.5 Endpoint drift

`/briefing` says recovery 70/Good. `/recovery/score` says 67/Good. Two source-of-truth numbers for the same metric. Whichever the redesign cites should be canonical; the other should disappear or proxy.

Likely cause: `/briefing` re-derives recovery from raw daily_context fields, while `/recovery/score` runs the canonical score formula. Pick one.

### 13.6 "Between phases" edge case is live right now

`active_phase: null`, `next_phase: "Riverdale prep"` starts May 11. That is the exact edge case the project flagged and **there's no surface treatment for it.** The Training tab and Goals sub-tab both show partial data; design against the live state, not a hypothetical.

### 13.7 Goal data is rich but unrendered

Each goal returns:
- `coaching_action` — what to do today
- `race_relevance` — tie-in to upcoming race
- `evidence_label` — `"strong"` / `"heuristic"`
- `expected_today` — projected value vs actual
- `last_update_label` — human-readable freshness
- `is_at_baseline` — flag for "haven't moved yet"
- `active_phase_role` — primary / secondary / maintenance

**Most of this never reaches the screen.** This is gold for a Strong-style detail view.

Reference: hit `GET /api/goals/dashboard` with the API key and inspect a single goal object.

### 13.8 Family pillar is a ghost in the data model

Two Mother's Day tasks. That's the entire family surface today. The project says 3 pillars, but the schema is **90% work + 10% training + 0% family**. Either:
- Add a family layer (events, milestones, contacts as first-class), or
- Accept that the home tab represents family via tasks + calendar only, and stop pretending otherwise.

### 13.9 Recovery components are pre-shaped for Whoop-style cards

Each component returns `{score, weight, detail}` where `detail` is already a one-sentence summary:
- Sleep: `"6h, quality 6/10"`
- Training load: `"TSB +64 (detraining) · CTL 291 / ATL 227"`

Drop these straight into context-paragraph cards in the recovery sheet. No additional formatting needed.

### 13.10 Yesterday: 0 tasks completed, full strength session today — briefing reports it flat

Briefing has the data ("yesterday: 0 done", "today: planned strength session") but there's no read on it. **A coach voice would react.** Empty yesterday + full today = either rest day or skipped — say so.

This is a copy/voice issue the data layer can't fix; the rendering layer needs a coach commentator that takes the raw signals and writes one human sentence.
